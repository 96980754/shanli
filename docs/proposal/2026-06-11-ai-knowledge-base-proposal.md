## 一、项目概述

### 1.1 项目背景

企业内部（客服、销售、行销团队）在遇到产品功能、技术参数和组合解决方案等问题时，严重依赖人工查找资料、询问产品部专职人员或大量产品培训，响应效率低、知识传承难。新员工入职后难以快速掌握公司知识，海外市场拓展也因语言障碍导致知识传递模糊且低效，丧失潜在商业机会。

此前研发团队已基于 Dify 做了初版 AI 知识库验证，发现三大核心问题：
1. **复杂表格解析弱**：对 Excel 等技术参数表解析效果差，行列错乱导致回答不精准
2. **多文档融合能力不足**：无法将多个产品资料有机融合，输出定制化组合方案
3. **使用频率低**：用户习惯依赖传统查表方式，不会主动使用知识库

本项目旨在构建一套 AI 原生的知识库系统，通过**知识图谱 + GraphRAG + 向量检索**的混合架构，实现企业知识的结构化存储、智能检索和高效利用。

### 1.2 核心目标

- **知识结构化**：将产品文档、技术参数表、FAQ、图片等非结构化数据转化为可检索的结构化知识
- **多文档融合**：基于知识图谱的实体关联能力，实现跨文档的知识整合与推理
- **精准问答**：基于 GraphRAG 技术提供即时、可溯源、高准确率的自然语言问答
- **持续进化**：通过用户反馈闭环和知识质量自维护机制，持续提升回答质量

### 1.3 技术特点

✓ **混合架构**：向量检索 + 知识图谱双引擎，兼顾快速上线与深度推理  
✓ **本体驱动**：基于业务领域构建专属知识本体，确保知识表示的规范性和一致性  
✓ **多模态抽取**：支持文档、表格、图片、流程图等多种数据源的知识抽取（渐进增强）  
✓ **增量更新**：通过冲突检测和实体对齐，实现知识的无缝增量更新  
✓ **自动维护**：定期执行知识质量检查，清理冗余、冲突和过期数据  
✓ **渐进交付**：三期递进，一期 MVP 快速上线，二期知识图谱增强，三期完善复杂场景

## 二、技术方案

### 2.1 系统架构

```
┌─────────────────────────────────────────────────────────────┐
│                     用户交互层                                │
│     Web 浏览器 (Vue 3)  |  企微 H5 插件  |  管理后台         │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                     应用服务层 (FastAPI)                      │
│    GraphRAG问答 | 向量检索 | 知识管理 | 用户管理 | 数据统计    │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                     知识处理层 (Celery + LLM)                 │
│   文档解析 | 实体关系抽取 | 数据清洗 | 知识维护 | 问答调优    │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                     数据存储层                                │
│  Neo4j图数据库 | Milvus向量库 | PostgreSQL | 文件存储 | Redis │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 核心设计决策

本方案在技术选型上遵循**「够用就好、按需升级」**原则，不做过度工程。以下六条是核心设计决策：

| # | 决策项 | 结论 | 理由 |
|---|--------|------|------|
| 1 | **NER/RE 模型** | ❌ 不训练专用模型 | 用 LightRAG 自带的 LLM 驱动抽取（few-shot 调用通义千问），省去 5000+ 实体标注成本 |
| 2 | **多智能体框架** | ❌ 一期不上 | 检索逻辑用确定性 Python 流水线即可，延迟可控、调试简单；二期如需复杂推理再评估 LangGraph |
| 3 | **Neo4j 版本** | 社区版起步 | 实体类型仅六七大类，社区版单机够用；企业版授权费（~22w/年）等用户量上来再评估 |
| 4 | **复杂表格解析** | 多模态 LLM 读图 | Qwen2.5-VL/Claude 视觉能力直接解析表格图片为 JSON，比传统 OCR+规则鲁棒得多，解决甲方核心痛点 |
| 5 | **表格辅助工具** | PP-StructureV2 | 百度 PaddleOCR 的表格识别模块，中文支持好，配合大模型做二次校正 |
| 6 | **图片向量化** | Chinese-CLIP（阿里达摩院） | 中文场景终端设备图片 + 中文标注，比原版 CLIP 更适配，二期上线 |
| 7 | **多智能体架构模式** | 单引擎 + function calling | 四Agent串行=4次LLM调用≈6s首token，不可接受。改为一个LLM调用+注册多个工具函数（retrieve/search等），工具纯函数执行<50ms，首token控制在≤1.5s。与Claude/GPT的tool use机制对齐 |

### 2.3 核心技术模块

#### 2.3.1 知识库管理模块

**功能描述**：支持知识库的创建、编辑、删除和分类管理，提供文档导入、权限控制、数据清洗和重复检测。

**技术实现**：
- 知识库单元化隔离，支持部门/项目组/个人级别的可见范围控制
- 文档解析：一期基于 **Unstructured.io**（开源 Apache 2.0）做基础解析，覆盖 Word/PDF/PPT 的标题、段落、简单表格抽取，表格转 HTML/Markdown 结构保留基本行列关系
- 数据清洗：LLM 自动将排版混乱文本重新排版 → 结构化输出 → 管理员确认后同步
- 重复检测：文档 hash + 语义相似度（余弦 > 0.95 判定为重复）

#### 2.3.2 多模态知识抽取模块

**功能描述**：从文档、表格、图片等数据源中抽取结构化知识（实体、属性、关系），渐进式构建知识图谱。

**技术实现**（三期递进）：

**一期（P0）— 简单表格 + 文本抽取**：
- 文档解析：Unstructured.io 提取标题/段落/简单表格，表格转为 Markdown 格式向量化
- 实体关系抽取：直接使用 **LightRAG** 自带的 LLM 驱动抽取模式（调用通义千问做 few-shot 抽取），不自研 NER/RE 模型
- 按文档类型配置抽取 Prompt 模板（产品手册用"产品-功能"模板，FAQ 用"问题-答案"模板）
- 抽取结果先入库待审，审核通过后写入知识库

**二期（P1）— 复杂表格 + 图片**：
- 复杂表格解析（甲方核心痛点）：**多模态 LLM（Qwen2.5-VL/Claude 视觉能力）** 直接读取表格图片，将其转为结构化 JSON
- 辅以 **PaddleOCR PP-StructureV2** 做表格结构识别（合并单元格、跨行跨列），配合大模型做二次校正
- 利用 LLM 的 reasoning 能力将行列映射为三元组 `(产品型号, 参数名, 参数值)`
- OCR 图片文字识别（PaddleOCR），建立图片-实体关联

**三期（P2）— 流程图 + 语音**：
- 流程图拓扑解析 → 流程三元组
- 语音输入转文本查询

#### 2.3.3 知识图谱构建与维护模块

**功能描述**：以 **LightRAG** 为核心引擎构建知识图谱（github 36.8k stars，EMNLP 2025），原生支持 Neo4j 图存储后端，自带增量更新和混合检索流水线。

**核心本体骨架**（在 LightRAG 的 Schema 配置中定义）：

```
实体类型：Product（产品）| Feature（功能）| Solution（方案）
          | Parameter（参数）| Document（文档）| Scenario（场景）

关系类型：has_feature | has_parameter | belongs_to | applies_to
          | describes | references | depends_on
```

**图数据库**：**Neo4j 社区版**起步，单机部署。实体类型仅六七大类，社区版性能完全够用。企业版集群和高阶图算法在用户量上来后再评估。

**增量更新流程**（LightRAG 原生支持）：
```
新文档 → LLM 抽取实体关系 → 与已有 KG 比对（LightRAG 内置去重）
  ├─ 新实体/关系 → 直接写入 Neo4j（增量，无需全量重建）
  ├─ 已有实体补充 → 属性融合
  └─ 冲突 → 标记待人工审核
```

**质量自维护**（三期实现）：
- 冗余检测：LightRAG 内置去重机制
- 冲突检测：按时间戳 + 来源优先级自动解决或标记待审
- 过时检测：超过有效期的三元组标记过期，排除出检索

#### 2.3.4 GraphRAG 智能问答模块

**功能描述**：以 **LightRAG** 作为 GraphRAG 引擎核心，提供混合检索（向量+图）与多模式问答。采用**单次LLM调用 + 函数式工具（function calling）**模式，检索/搜索等操作为纯函数（无需LLM参与，<50ms），仅在最终答案生成时调用一次LLM，确保首token延迟≤1.5s。

**检索问答流程**（非多智能体串行，而是单引擎 + tools）：

```
用户问题
    ↓
LLM（一次调用，tools=[retrieve, bm25_search, graph_search, ...]）
    │
    ├─ 自动分析问题意图 → 选择调用哪个/哪些工具
    ├─ 工具执行（纯函数，无LLM参与）
    │   ├─ retrieve(query, kb_id)        → Milvus语义检索 Top-30      <100ms
    │   ├─ bm25_search(query, kb_id)     → BM25精确匹配 Top-20         <50ms
    │   ├─ graph_search(entities)        → Neo4j图检索（二期）         <200ms
    │   └─ rerank(results)               → 精排 Top-10                 <10ms
    ├─ 工具结果返回LLM上下文
    └─ 生成答案 + 溯源（单次生成）                                       <1.2s
                                                   ↓
                                           首 token ≤ 1.5s ✅
```

**对比传统四智能体串行方案**：

| 方案 | LLM调用次数 | 首token延迟 | 延迟根因 |
|------|:----------:|:----------:|---------|
| 四智能体串行（调度→检索→推理→融合） | 4次 | ~6s | 每次LLM调用1.5s，串行累加 |
| 单引擎+function calling（✅ 本方案） | **1次** | **≤1.5s** | 检索工具纯函数<50ms，不增加LLM串行次数 |
- 检索内容相关度低于阈值 → 拒答，返回预设话术
- 用户反馈"没帮助"→ 记录 → 管理员调优
- 最新版本检索加权，优先展示最新信息

**性能指标**：
- 问答准确率 ≥ 80%
- 检索响应 <500ms，首 token <1.5s（LightRAG 轻量配置秒级以内）
- 引用正确率 ≥ 95%

#### 2.3.5 系统管理与数据闭环

**功能描述**：用户管理、角色权限控制、问答数据统计和知识质量反馈闭环。

**权限模型**：
```
系统管理员（产品经理）— 全权限
  ├─ 知识库管理员（营销运作/客服负责人）— 知识管理、审核、调优
  └─ 知识查询者（客服/销售/行销）— 提问、浏览、反馈
```

**数据反馈闭环**：
```
员工提问 → AI回答 → 评价反馈（👍/👎）
  ↓
系统记录低分答案 → 管理员每周审阅
  ├─ 文档有误 → 更新文档 → 重新同步
  ├─ 文档缺失 → 撰写 FAQ → 追加到知识库
  └─ 模型问题 → 调优 prompt / 添加问答映射
  ↓
知识库更新 → 下次相同/相似问题自动改善
```

## 三、技术栈与系统环境

### 3.1 核心技术栈

| 技术层 | 技术选型 | 说明 |
|-------|---------|------|
| **图数据库** | Neo4j 社区版 5.x | 单机起步，实体量不大，社区版够用 |
| **向量数据库** | Milvus 2.x | 开源高性能向量检索引擎，支撑 50 万+ 文档片段 |
| **GraphRAG 引擎** | LightRAG（github 36.8k ⭐） | 增量更新 + 混合检索 + Neo4j 原生支持，开箱即用 |
| **大语言模型** | 通义千问 / GPT-4 / Claude | 知识抽取、问答生成；复杂表格用多模态版本；**注意：如客户要求私有化部署或数据不出境，则只能选通义千问（或私有化部署 Qwen2.5 开源模型），GPT-4 和 Claude 不可用** |
| **嵌入模型** | BGE-Large-ZH（智源），英文 >30% 自动切换 M3E | 中文语义检索业界领先；当查询中英文占比超过 30% 时自动切换 M3E 多语言模型，保障双语场景检索效果 |
| **稀疏检索** | rank_bm25 | 一期轻量实现，与语义检索互补 |
| **重排序（一期）** | 规则词重叠 | 零延迟，够用 |
| **重排序（二期）** | BGE-Reranker-v2-m3（智源） | 交叉注意力精排，提升 10-15 百分点 |
| **文档解析** | Unstructured.io（一期） | 开源 Apache 2.0，Word/PDF/PPT 解析 |
| **表格解析（二期）** | PP-StructureV2（结构识别）→ Qwen2.5-VL（内容校正） | 管道分工：PP-StructureV2 先做表格结构识别（切出行列框架、检测合并单元格），再将结构化的表格图片送入 Qwen2.5-VL 做语义理解和内容校正（修正 OCR 错字、理解层级关系、输出结构化 JSON），两个角色不可互换 |
| **图像向量化（二期）** | Chinese-CLIP（阿里达摩院） | 中文场景适配，以图搜图 |
| **后端框架** | Python FastAPI | API 服务和业务逻辑 |
| **前端框架** | Vue.js 3 + Element Plus | 用户交互界面 |
| **任务调度** | Celery + Redis | 异步任务和定时维护 |
| **监控日志** | Prometheus + Grafana | 系统监控和性能分析 |

## 八、项目风险与应对

| 风险点 | 等级 | 说明 | 应对措施 |
|--------|------|------|---------|
| **复杂表格解析精度不足** | 中（核心痛点） | Excel 表格含合并单元格、跨行跨列，传统 OCR+规则方案易出错 | 一期 Unstructured.io 转 Markdown 兜底；二期多模态 LLM（Qwen2.5-VL）直接读图转 JSON，辅以 PP-StructureV2 |
| **多文档融合推理效果不佳** | 中 | AI 检索融合效果不稳定，组合方案回答不完整 | LightRAG 图检索（二期）通过实体关系关联跨文档知识 |
| **LLM API 服务不稳定** | 低 | 依赖云端大模型 API，网络波动或限流影响可用性 | 支持多厂商切换（Qwen/GPT-4/Claude），设置降级策略 |
| **首 token 延迟超标** | 中 | 多级管道可能导致累积延迟 >1.5s | 采用单次LLM调用+纯函数工具模式，避免串行LLM调用；检索工具<50ms不增加LLM串行次数 |
| **用户使用意愿低** | 中 | 习惯查表，不愿主动使用知识库 | 一期与企微深度集成，降低使用门槛；配合反馈闭环持续提升回答质量 |
| **知识库优化是长期工程** | 高 | 需要持续投入人力维护和调优，否则效果衰减 | 前期开发+产品经理投入，后期业务负责人（营销运作/客服）持续维护 |
