# 自助注册与权限文档工作台设计

**日期：** 2026-07-14  
**目标：** 支持人工筛选的真实文件由管理员上传，普通用户自助注册后由管理员授予知识库权限与知识视图规则；用统一的深色运营工作台完成浏览、下载、授权和问答验收。

---

## 1. 范围

本阶段包含：

1. 开放普通用户注册；
2. 数据库模式使用安全密码哈希保存与验证注册用户；
3. 注册用户默认无知识库权限；
4. 管理员人工创建知识库、上传人工筛选的真实文件并填写元数据；
5. DOCX/PDF 解析并参与问答；PPTX/XLSX 保存、可下载、明确标记为不支持内容解析；
6. 管理员从工作台选择已注册用户，授予/撤销知识库权限并维护知识视图规则；
7. `/documents` 三栏权限工作台；
8. 登录、注册、管理台、问答页和文档页共享深色专业运营工作台视觉系统；
9. 自动化测试及人工上传的真实验收流程。

本阶段不包含：

- ZIP 解压、批量初始化、固定样本映射或预置用户；
- PPTX/XLSX/OCR/VLM 内容解析；
- 自动授予新注册用户任何知识库权限；
- React、前端构建工具、完整角色/组织体系或多租户改造。

---

## 2. 账号、注册和认证

### 2.1 注册 API

新增：

```http
POST /api/auth/register
Content-Type: application/json
```

请求：

```json
{
  "username": "alice",
  "password": "correct-horse",
  "password_confirmation": "correct-horse"
}
```

规则：

- `username` 去除首尾空格后必须为 3–64 个字符；只允许字母、数字、下划线、连字符和点；
- 密码长度至少 8；
- `password_confirmation` 必须完全相同；
- 用户名必须唯一；
- 成功创建普通用户和默认 `普通用户` role，返回 `201` 与用户 ID、用户名；
- 注册不创建知识库权限或知识视图规则；
- 重复用户名返回 `409 Username already exists`；格式或密码校验失败返回 `422`；
- 不返回密码哈希或任何密码字段。

### 2.2 密码策略

数据库模式中的新注册用户使用 `bcrypt` 哈希保存密码。登录通过 `bcrypt.checkpw()` 验证。新用户、管理员和任何数据库账户都使用同一验证路径；不使用数据库 `password_hash` 与明文密码的相等比较。

内存模式保持既有 `admin/admin` 以维持 MVP/既有测试，不提供持久化注册。

数据库管理员的初始化不属于开放注册流程：部署者应通过显式管理/迁移方式创建首个管理员账户。为支持本地体验，注册接口不允许客户端设置管理员身份；管理员权限仍来自知识库 `can_grant`。

### 2.3 登录 UI

`/login` 采用登录/注册双面板或切换式卡片：

- 登录：用户名、密码、提交；
- 注册：用户名、密码、确认密码、提交；
- 成功注册提示“账号已创建，请登录；管理员授权后才能访问知识库”；
- 不预填账号或密码，不展示演示凭据；
- 登录成功进入 `/documents`；未认证访问工作台时跳转 `/login`。

## 3. 人工上传与文档处理

管理员（有 `can_upload`）在管理台选择人工筛选的真实文件，填写：部门、产品线、visibility、密级、标签、scope、文档类型、产品和优先级。

| 文件格式 | 保存 | 解析状态 | 问答 | 下载 |
|---|---|---|---|---|
| DOCX、PDF | 是 | `parsed` | 生成 ContentBlock / DocumentChunk | 可用 |
| PPTX、XLSX | 是 | `stored_unsupported` | 不进入索引 | 可用 |

数据库文档列表和详情统一返回：

```json
{
  "status": "stored_unsupported",
  "parse_available": false,
  "parse_status_label": "仅可下载（暂不支持内容解析）",
  "download_available": true
}
```

`parsed` 返回 `parse_available=true` 与“已解析，可用于问答”。原文件缺失的历史文档保持 `download_available=false`。不支持格式不是错误：不创建无用 ParseTask，不回滚文件，不计入问答 chunk。

下载继续通过服务端 `DocumentAccessService` 授权并在成功后写 `download_document` 审计记录。

## 4. 授权和工作台

### 4.1 默认权限

新注册用户：

```text
无 KnowledgeBasePermission
无 KnowledgeViewRule
```

其登录后可访问 `/documents`，但 `/api/kb` 返回空集合；页面显示无权限空状态和管理员联系提示，不能推断任何知识库或文档存在。

### 4.2 管理员授权流程

管理员在选择知识库后，从“已注册用户”列表选择用户：

1. 写入 `can_view`、`can_upload`、`can_delete`、`can_grant`；
2. 当 `can_view=true` 时，可设置部门、产品线、visibility、最大密级组成的知识视图规则；
3. 撤销 `can_view` 后同时删除该用户的知识视图规则，防止留下不生效的孤立规则；
4. 普通用户刷新页面或重新登录后，根据服务端有效规则得到新的可见集合。

管理员用户列表新增只读 API：

```http
GET /api/users
```

要求当前用户至少对一个知识库具备 `can_grant`；仅返回 `id`、`username` 和 role 名称，不返回哈希。原有 `GET/PUT/DELETE /api/kb/{kb_id}/permissions/{user_id}` 与知识视图规则 API 继续作为唯一的知识库授权写接口。

### 4.3 `/documents` 工作台

桌面端采用三栏、窄屏单列：

- 顶栏：产品标识、当前账号、导航、退出；
- 左栏：可访问知识库、状态统计、解析状态/visibility/产品筛选；
- 中栏：仅展示后端已授权返回的文档，显示格式、元数据、解析和下载状态；
- 右栏：详情、元数据、解析统计与下载；
- 管理员区：仅在当前账户拥有选中知识库 `can_grant` 时出现，加载 `/api/users` 并完成权限/规则编辑。

客户端筛选只在已授权文档中发生，不承担安全职责。状态标签使用文字和颜色双重表达。

## 5. 视觉系统

新增单一 `backend/app/static/app.css`，在 `/login`、`/admin`、`/qa`、`/documents` 引入。

风格为深色专业运营工作台：深蓝灰分层表面、青蓝强调色、克制的成功/提醒/危险状态色、统一圆角和焦点环；提供 `.app-shell`、`.topbar`、`.panel`、`.button`、`.status-badge`、`.empty-state`、表单、表格和响应式布局。保持原生 HTML/JS，不引入框架。

## 6. 测试和验收

自动化测试：

1. 注册成功、密码长度、确认密码、用户名格式和重复用户名；
2. 注册用户密码被 bcrypt 哈希，正确密码可登录、错误密码不可登录；
3. 新注册用户无知识库权限；
4. `/api/users` 只允许 `can_grant` 用户，且不泄露密码字段；
5. 管理员授权/撤销用户后，知识库与文档可见范围正确变化，撤销时规则被删除；
6. PPTX/XLSX 为 `stored_unsupported`、可下载、没有 chunk；DOCX/PDF 为 `parsed`；
7. 列表/详情的解析状态字段与下载权限；
8. 登录/注册、空权限状态、工作台和管理员用户选择器的前端壳；
9. 现有下载、审计、知识视图和问答回归测试。

人工验收：

1. 以部署者创建的管理员账户登录；
2. 创建知识库并人工上传已筛选的 DOCX/PDF/PPTX/XLSX，填写正确元数据；
3. 注册普通用户，确认工作台为空；
4. 管理员在用户列表中选择该用户，授予 `can_view` 并设置规则；
5. 用户重新登录，确认只看见规则允许的文档；
6. 下载有权限的 DOCX/PDF/PPTX/XLSX，确认文件名与内容；
7. 撤销权限后确认用户再次无访问权限；
8. 检查下载审计。

---

## 7. 成功标准

- 不需要 ZIP 自动导入或固定演示用户；
- 普通用户可开放注册，但默认无知识库访问；
- 密码以 bcrypt 哈希存储和验证；
- 管理员可在工作台对已注册用户完成授权、规则配置和撤销；
- 手动上传的 DOCX/PDF 可问答，PPTX/XLSX 保留可下载；
- 四个页面形成一致的深色运营工作台风格；
- 权限、下载和审计均由后端授权路径保证。
