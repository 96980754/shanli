# 知识库管理员后台 v1 设计

**日期：** 2026-07-07

**目标：** 在当前已具备登录、知识库 CRUD、文档上传、文档详情、前端管理台壳的基础上，建设一个真正可操作的知识库管理员后台 v1，支持用户级可累加权限控制，以及知识库与文档的最小可操作管理闭环。

---

## 1. 设计范围

本阶段聚焦知识库管理员的核心工作流：

1. 管理知识库（新建、查看、删除）
2. 管理文档（上传、查看、删除）
3. 管理知识库用户权限（查看、上传、删除、授权四项可累加权限）
4. 在 Web 管理台完成以上操作

本阶段**不包含**：

- 系统管理员后台（用户全生命周期管理、角色体系管理）
- 问答日志、反馈统计、知识缺口分析
- 联网补知识闭环
- 图片找终端、库存系统对接、语音输入、移动端 App
- 流程审批、组织树、部门继承授权、审计日志查询

---

## 2. 角色与权限模型

### 2.1 当前涉及角色

本阶段只明确 2 类系统参与者：

- **知识库管理员**：维护知识库、文档和知识库内用户权限
- **知识查询者**：在被授权的知识库内进行浏览、检索、问答

系统管理员能力留到后续阶段。

### 2.2 权限作用域

权限是**按知识库生效**的，不是全局生效。

也就是说：

- 同一个用户在 A 知识库可以有“查看 + 上传”
- 在 B 知识库可以只有“查看”
- 在 C 知识库可以有“查看 + 上传 + 删除 + 授权”

### 2.3 权限项

每个用户在每个知识库下，支持 4 个可累加权限：

- `can_view`
- `can_upload`
- `can_delete`
- `can_grant`

说明：

- `can_view`：可查看知识库、文档列表、文档详情，并可进行知识查询
- `can_upload`：可上传文档
- `can_delete`：可删除文档
- `can_grant`：可查看与编辑知识库权限，且阶段 1 中也用于删除知识库

### 2.4 默认授权规则

甲方已确认：

> 新建知识库后，**创建者自动拥有 查看 / 上传 / 删除 / 授权 4 项权限**。

这是阶段 1 的关键规则，因为否则创建者建完知识库后将无法继续管理。

### 2.5 未授权用户可见性规则

阶段 1 采用严格可见性策略：

- 未被授予 `can_view` 的用户，看不到该知识库
- 也无法访问该知识库的详情、文档列表、文档详情

该规则适用于：

- 知识库列表
- 知识库详情
- 文档列表
- 文档详情

## 3. 数据模型设计

### 3.1 用户来源

甲方已确认：

> 用户来源先基于现有 `users` 表。

阶段 1 不新建独立用户中心，不做外部账号打通，只在现有用户表基础上实现知识库内授权。

### 3.2 权限数据结构

建议以“知识库用户权限表”表达一条授权记录。每条记录包含：

- `kb_id`
- `user_id`
- `can_view`
- `can_upload`
- `can_delete`
- `can_grant`

如果当前代码中的 `kb_permissions` 模型不足以表达该结构，则在现有模型基础上补齐字段，而不是另外设计单值角色枚举。

### 3.3 设计理由

采用布尔字段，而不是单字符串角色值，有以下好处：

1. 直接支持“可累加多权限”
2. 前端复选框保存简单
3. 后端鉴权直观
4. 未来新增权限项时可平滑扩展

## 4. 核心后端接口设计

### 4.1 权限查询接口

#### GET `/api/kb/{kb_id}/permissions`

返回某知识库下的用户权限列表。

**返回示例：**

```json
{
  "items": [
    {
      "user_id": 1,
      "username": "admin",
      "can_view": true,
      "can_upload": true,
      "can_delete": true,
      "can_grant": true
    }
  ]
}
```

**权限要求：**
- 当前用户必须拥有该知识库的 `can_grant`

### 4.2 权限设置接口

#### PUT `/api/kb/{kb_id}/permissions/{user_id}`

为某用户设置该知识库下的多选权限。

**请求示例：**

```json
{
  "can_view": true,
  "can_upload": true,
  "can_delete": false,
  "can_grant": false
}
```

**语义：**
- 用请求体覆盖该用户在该知识库下当前权限状态

**权限要求：**
- 当前用户必须拥有该知识库的 `can_grant`

### 4.3 权限移除接口

#### DELETE `/api/kb/{kb_id}/permissions/{user_id}`

移除某用户在该知识库下的权限记录。

**权限要求：**
- 当前用户必须拥有该知识库的 `can_grant`

### 4.4 现有接口的权限升级

#### 知识库列表 / 详情
- 要求 `can_view`
- 未授权知识库不出现在列表中

#### 文档列表 / 详情
- 要求 `can_view`

#### 文档上传
- 要求 `can_upload`

#### 文档删除
- 要求 `can_delete`

#### 知识库删除
- 阶段 1 先要求 `can_grant`

## 5. 管理台页面设计

### 5.1 页面目标

将当前 `/admin` 从“静态壳”升级为“知识库管理员工作台 v1”。

### 5.2 页面布局

采用三段式布局：

#### 左侧：知识库列表区
展示：
- 知识库名称
- 描述
- 文档数量

操作：
- 切换知识库
- 新建知识库
- 删除知识库（仅有 `can_grant` 时显示）

#### 中间：文档管理区
拆分为 3 段：

1. **上传区**
   - 单文件选择框
   - 上传按钮
   - 上传结果提示
2. **文档列表区**
   - 文档标题
   - 文件类型
   - 状态
   - 删除按钮（仅有 `can_delete` 时显示）
3. **文档详情区**（放在文档列表下方）
   - 标题
   - 文件类型
   - 状态
   - `block_count`
   - `chunk_count`

上传成功后，系统应：
- 刷新文档列表
- 自动选中新上传文档
- 自动加载并显示详情

删除文档后，系统应：
- 刷新文档列表
- 若删除的是当前选中文档，则清空详情区

#### 右侧：权限与操作区
包含：
- 当前知识库权限列表
- 授权编辑表单
- 删除确认区
- 成功/失败提示区

### 5.3 删除确认交互

延续此前已确认的交互方式：

- 不使用浏览器原生 `confirm`
- 使用页面内确认面板
- 支持：
  - 删除文档确认
  - 删除知识库确认
  - 移除用户权限确认（可选复用）

### 5.4 授权交互

建议在权限区提供：

- 用户下拉框（来自现有 `users` 表）
- 4 个复选框：查看 / 上传 / 删除 / 授权
- 保存按钮

保存后调用：
- `PUT /api/kb/{kb_id}/permissions/{user_id}`

并立即刷新权限列表。

### 5.5 文档管理深化边界

本阶段文档管理深化**不做**：

- 批量上传
- 原文预览
- 文本片段预览
- PDF / 图片可视化预览
- 拖拽上传

目标是先把“上传 → 列表 → 详情 → 删除”闭环打稳。

## 6. 页面行为与状态流转

### 6.1 登录后加载

1. 读取本地 `session_token`
2. 调用 `/api/auth/me`
3. 拉取当前用户可见知识库列表
4. 默认选中第一个知识库
5. 加载：
   - 文档列表
   - 权限列表（若当前用户有 `can_grant`）
   - 当前选中文档详情（若已有选中项）

### 6.2 新建知识库

1. 用户填写最小表单（名称、描述、可见范围）
2. 提交创建知识库
3. 后端自动为创建者授予 4 项权限
4. 前端刷新知识库列表并自动选中新建知识库

### 6.3 上传文档

1. 用户选择单个文件
2. 点击上传按钮
3. 调用上传接口
4. 上传成功后：
   - 刷新文档列表
   - 自动选中新上传文档
   - 调用详情接口加载详情
   - 页面提示“上传成功”

### 6.4 删除文档

1. 用户点击删除
2. 页面内显示确认区
3. 用户确认后调用删除接口
4. 删除成功后：
   - 刷新文档列表
   - 若删除的是当前选中文档，则清空详情区
   - 页面提示“删除成功”

### 6.5 管理权限

1. 用户查看当前知识库权限列表
2. 通过授权表单给指定用户勾选多权限
3. 提交后刷新权限列表
4. 如需移除授权，调用删除权限接口

### 6.6 删除知识库

1. 用户点击知识库删除
2. 页面内显示确认区
3. 调用知识库删除接口
4. 成功后刷新知识库列表并切换到下一个可见知识库或空状态

### 6.7 前端最小状态

建议维护：

- `authProfile`
- `knowledgeBases`
- `activeKbId`
- `documents`
- `permissions`
- `selectedDocumentId`
- `selectedDocumentDetail`
- `uploading`
- `pendingDeleteDocument`
- `pendingDeleteKnowledgeBase`
- `message`

这样可以保证：
- 上传后自动选中并打开新文档详情
- 删除当前文档后正确清空详情
- 切换知识库后正确重置详情区

## 7. 测试策略

### 7.1 认证与会话测试

验证：
- 登录后能识别当前用户
- `/api/auth/me` 返回当前用户信息
- 权限接口与管理接口均可复用同一认证头

### 7.2 权限接口测试

覆盖：
- 未登录访问权限接口返回 `401`
- 无 `can_grant` 返回 `403`
- 有 `can_grant` 可查看权限列表
- 可为某用户设置任意多选权限组合
- 删除权限后该用户失去访问能力

### 7.3 业务鉴权测试

覆盖：
- 无 `can_view` 看不到知识库与文档
- 有 `can_view` 可查看
- 有 `can_upload` 才能上传
- 有 `can_delete` 才能删文档
- 有 `can_grant` 才能改权限和删知识库

### 7.4 前端壳测试

最小验证：
- `/admin` 页面包含知识库区、文档区、权限区、删除确认区
- 页面包含上传区、文档详情区、权限编辑区、页面消息区
- 静态资源路径可访问
- 页面骨架仍返回 `200`

### 7.5 当前阶段不做的前端测试

本阶段不引入浏览器自动化来验证点击链路，只验证：
- 页面结构到位
- 后端接口闭环稳定

如果后续需要验证真实点击、上传、删除流程，再单独引入浏览器级测试。

## 9. 验收标准

若满足以下条件，则阶段 1：知识库管理员后台 v1 视为完成：

1. 管理员可创建、查看、删除知识库
2. 管理员可上传、查看、删除文档
3. 管理员可查看、设置、移除知识库用户权限
4. 一个用户可在同一知识库下拥有多个可累加权限
5. 未授权用户看不到该知识库
6. 管理台可完成知识库列表切换、单文件上传、文档详情查看、文档删除、权限编辑
7. 上传成功后自动选中新文档并展示详情
8. 删除当前选中文档后会清空详情区
9. 新增测试与完整回归通过
