# 知识库 / 知识图谱权限与元数据过滤设计

**日期：** 2026-07-09

**目标：** 在现有知识库级用户权限模型基础上，设计一套可演进的“元数据硬过滤 + 标签辅助分类”方案，用于支撑不同用户在同一知识库下拥有不同知识视图，并为后续图谱检索提供统一权限边界。

---

## 1. 背景与问题定义

当前系统已经完成了：

- 知识库级用户权限控制；
- 文档上传与解析；
- RAG 问答；
- 问答记录与知识缺口闭环。

当前权限边界是：

> **先判断用户是否能进入某知识库，再只在该知识库中执行问答与管理操作。**

这意味着当前“不同用户不同知识视图”的粒度是：

> **知识库级**

而不是：

- 文档级；
- chunk 级；
- 图谱实体 / 关系级。

随着后续继续接：

- 更大的文档规模；
- 更复杂的检索；
- 图谱检索 / GraphRAG；
- 多角色、多部门、多产品线知识分发；

需要把权限设计从“知识库能否进入”扩展到：

> **进入知识库后，当前用户还能看到哪些文档、哪些 chunk、哪些图谱子图。**

---

## 2. 当前已实现权限模型

### 2.1 知识库级权限

当前权限表：`kb_permissions`

字段：

- `kb_id`
- `user_id`
- `can_view`
- `can_upload`
- `can_delete`
- `can_grant`

语义：

- `can_view`：可查看知识库、文档、问答；
- `can_upload`：可上传文档；
- `can_delete`：可删除文档；
- `can_grant`：可查看与修改知识库权限；当前阶段也承担部分管理员能力。

### 2.2 当前 RAG 的权限边界

当前问答链路先校验：

1. 用户是否登录；
2. 用户是否对目标知识库拥有 `can_view`；
3. 只有校验通过，才会执行 `/api/qa/ask/sync` 检索与问答。

因此当前知识视图差异来自：

> **不同用户可访问的知识库集合不同。**

这已经足以支撑一期功能，但不足以支撑同一知识库内的细粒度知识隔离。

---

## 3. 设计原则

本设计采用：

> **元数据做硬过滤，标签做辅助分类。**

### 3.1 元数据硬过滤

元数据用于真正的权限边界控制。

即：

- 文档是否进入当前用户的可检索集合；
- chunk 是否进入当前用户的可检索集合；
- 图谱实体 / 关系是否进入当前用户的可检索子图。

### 3.2 标签辅助分类

标签用于：

- 业务分类；
- 检索增强；
- 前端筛选；
- 运营分析；

但：

> **标签不能作为唯一权限依据。**

---

## 4. 文档层模型设计

### 4.1 建议新增的元数据字段

建议在文档级先引入最少 4 个结构化字段：

- `department`
- `product_line`
- `visibility`
- `security_level`

### 4.2 字段语义

#### `department`

文档所属业务域，例如：

- 售后
- 交付
- 销售支持
- 研发支持

#### `product_line`

文档所属产品线，例如：

- P368
- MCSTARS
- X1000

#### `visibility`

建议固定枚举：

- `public`
- `internal`
- `restricted`

#### `security_level`

建议数值化，便于范围过滤：

- `1`：普通内部资料
- `2`：部门受限资料
- `3`：高敏资料

### 4.3 标签字段

建议单独保留：

- `tags: list[str]`

示例：

```json
{
  "tags": ["FAQ", "报警", "安装", "高频问题"]
}
```

标签不直接承担“用户能不能看”的权限决策，只承担“在可见内容里如何筛和如何组织”的辅助作用。

---

## 5. 用户知识视图画像设计

知识库权限决定“能不能进入知识库”，但进入后还需要一份“知识视图过滤画像”。

建议表达为：

```json
{
  "allowed_departments": ["售后"],
  "allowed_product_lines": ["P368", "MCSTARS"],
  "allowed_visibility": ["public", "internal"],
  "max_security_level": 2
}
```

语义：

- 用户必须先拥有该知识库的 `can_view`；
- 在此基础上，再按画像决定可见文档集合。

这套画像建议在后续阶段作为“知识视图规则”落库，不与 `kb_permissions` 强行塞进一张表，以保持职责清晰。

---

## 6. RAG 检索链路改造建议

目标：

> **先过滤，再检索**

而不是“先全量检索，再把结果过滤掉”。

### 6.1 推荐链路

```text
用户发起问答
→ 校验用户对知识库拥有 can_view
→ 读取该用户在该知识库下的知识视图画像
→ 按文档元数据硬过滤，得到可见文档集合
→ 加载这些文档的 chunks
→ 在可见 chunks 上执行 BM25 / 向量检索 / rerank
→ 如需要，再在可见集合中按标签做辅助过滤
→ 返回答案和来源
```

### 6.2 设计理由

这样做有 4 个优势：

1. 权限边界更安全；
2. 检索范围更小，性能更好；
3. 后续接向量库时可直接映射到 metadata filter；
4. 后续接图谱检索时可复用同一套文档过滤边界。

---

## 7. 图谱权限设计

### 7.1 当前结论

> **知识图谱权限不单独设计节点级 ACL，而是继承文档可见性。**

即：

- 文档可见 → 文档抽取出的实体 / 关系可见；
- 文档不可见 → 文档抽取出的实体 / 关系不可见。

### 7.2 为什么不做节点级 ACL

原因：

1. 节点 / 边 ACL 复杂度高；
2. 实体和关系往往来自文档，不应脱离文档单独授权；
3. 一期 / 二期早期更需要稳定、统一、可解释的权限边界；
4. 文档继承模型更适合后续审计和追溯。

### 7.3 GraphRAG 推荐链路

```text
用户发起图谱检索
→ 校验知识库 can_view
→ 读取知识视图画像
→ 先过滤出可见文档集合
→ 由可见文档推导可见实体 / 关系子图
→ 只在可见子图上执行图谱检索
→ 返回图谱结果
```

这样 RAG 和 GraphRAG 的权限边界保持一致。

## 8. 后台授权与配置建议

建议将后台权限配置拆成两层：

## 8.1 第一层：知识库进入权限（当前已实现）

保留现有：

- 查看
- 上传
- 删除
- 授权

## 8.2 第二层：知识视图过滤范围（后续新增）

建议在管理员后台增加“知识视图规则配置”区域，允许按用户配置：

- 可见部门（多选）
- 可见产品线（多选）
- 可见范围（多选）
- 最大密级（下拉）
- 辅助标签（可选，非权限）

这样管理员可以表达：

> 同一知识库下，A 用户能看售后 + P368 + 2 级以内资料，B 用户只能看交付 + MCSTARS + 1 级资料。

---

## 9. 实施优先级

建议分阶段推进，不要一次性做完。

### 阶段 1：定义文档元数据字段

补齐：

- `department`
- `product_line`
- `visibility`
- `security_level`
- `tags`

目标：先把数据模型和文档导入链路补齐。

### 阶段 2：检索前元数据硬过滤

改造：

- `DbChunkLoader`
- 问答检索加载逻辑

目标：只加载当前用户可见文档的 chunks。

### 阶段 3：后台配置知识视图规则

新增后台配置界面与数据存储，允许管理员按用户配置：

- allowed departments
- allowed product lines
- allowed visibility
- max security level

### 阶段 4：图谱继承文档过滤

在接入 Neo4j / GraphRAG 时，让图谱检索只查询“可见文档派生的子图”。

---

## 10. 最终规则总结

建议把规则固定成以下一句话，作为后续团队和甲方对齐的标准描述：

> **知识库权限决定用户是否能进入某知识库；文档元数据决定用户进入知识库后能看到哪些内容；标签仅用于业务分类和辅助筛选，不作为唯一权限依据；图谱可见性继承文档可见性。**

---

## 11. 建议结论

### 当前阶段

继续保留：

- 知识库级用户权限模型；
- `can_view/can_upload/can_delete/can_grant` 四项权限。

### 下一阶段

优先补：

- 文档元数据字段；
- RAG 检索前元数据硬过滤；
- 管理员后台的知识视图过滤配置。

### 图谱阶段

不要单独做节点 / 边 ACL，统一采用：

> **图谱继承文档可见性**

这样能保证：

- 实现复杂度可控；
- 权限边界一致；
- 审计可解释；
- 未来扩展到 GraphRAG 时不需要推翻现有模型。
