# 文档元数据字段设计与落库设计

**日期：** 2026-07-09

**目标：** 为后续“元数据硬过滤 + 标签辅助分类”的权限演进方案打基础，先把文档级元数据字段正式落到系统里，并让上传、详情和管理台都能感知这些字段。

---

## 1. 设计范围

本阶段只做：

1. 文档模型新增元数据字段；
2. 数据库模式上传接口支持写入元数据；
3. 文档详情接口返回元数据；
4. 管理台文档详情区展示元数据；
5. 测试、文档和回归同步更新。

本阶段**不做**：

- 用户知识视图规则表；
- 检索前元数据过滤；
- 图谱权限过滤；
- 标签搜索；
- 元数据批量编辑；
- 标签管理器；
- 高级校验工作流。

---

## 2. 字段设计

结合前一份权限演进设计，当前阶段先落最小 5 个字段：

### 2.1 硬过滤字段

- `department: str`
- `product_line: str`
- `visibility: str`
- `security_level: int`

### 2.2 辅助分类字段

- `tags: str`

当前阶段 `tags` 先用逗号分隔字符串存储，例如：

```text
FAQ,报警,安装,高频问题
```

后续如果需要更强的标签管理能力，再演进为 JSON 数组或独立标签表。

---

## 3. 模型层设计

建议在 `Document` 模型上新增：

- `department`，默认空字符串；
- `product_line`，默认空字符串；
- `visibility`，默认 `internal`；
- `security_level`，默认 `1`；
- `tags`，默认空字符串。

设计原则：

1. 当前阶段优先保证可落库、可回显；
2. 未传值时有稳定默认值；
3. 不阻塞原有上传链路；
4. 后续做元数据过滤时无需再做大规模 schema 改动。

---

## 4. 上传接口设计

### 4.1 接口范围

复用现有接口：

```http
POST /api/kb/{kb_id}/documents/upload
```

### 4.2 请求扩展

当前仍使用 `multipart/form-data`，除 `file` 外，增加可选表单字段：

- `department`
- `product_line`
- `visibility`
- `security_level`
- `tags`

### 4.3 默认值策略

若上传时未传：

- `department = ""`
- `product_line = ""`
- `visibility = "internal"`
- `security_level = 1`
- `tags = ""`

这样可以兼容旧的上传测试和旧页面。

### 4.4 当前阶段输入约束

为了先落地，本阶段不做复杂校验，只做最少约束：

- `visibility` 若为空则回退 `internal`；
- `security_level` 若为空则回退 `1`；
- 其余字符串字段允许为空。

高级枚举校验放到下一阶段再补。

---

## 5. 文档详情接口设计

复用现有接口：

```http
GET /api/kb/{kb_id}/documents/{doc_id}
```

当前返回体在原有字段基础上增加：

- `department`
- `product_line`
- `visibility`
- `security_level`
- `tags`

数据库模式和内存模式都应返回这些字段，以保持前端结构一致。

对于内存模式：

- 可以跟随上传时一并记录；
- 若没有则回默认值。

---

## 6. 管理台展示设计

在 `/admin` 文档详情区新增几行：

- 部门
- 产品线
- 可见范围
- 密级
- 标签

展示目标不是编辑这些字段，而是：

> **让管理员能看见文档已经具有结构化元数据。**

这一步只做回显，不做编辑界面。

---

## 7. 与权限演进的衔接

这一阶段不是最终权限实现，而是下一阶段的底座。

后续将基于这些字段继续推进：

### 第一步（当前阶段）
- 元数据字段落库；
- 上传接口支持写入；
- 详情和后台可见。

### 第二步（后续阶段）
- 增加用户知识视图规则；
- 例如：允许部门、允许产品线、最大密级。

### 第三步（后续阶段）
- RAG 检索前先按元数据硬过滤文档；
- 再加载可见 chunk；
- 再执行 BM25 / 向量检索。

### 第四步（后续阶段）
- GraphRAG / 图谱检索继承文档可见性；
- 不单独做节点 / 边 ACL。

---

## 8. 测试策略

### 8.1 模型 / schema 测试

验证：

- `Document` 模型新增字段；
- 默认值可持久化；
- 数据库可正确读回。

### 8.2 API 测试

新增或扩展测试覆盖：

1. 上传文档时传入元数据，返回体和详情接口可读到；
2. 上传文档时不传元数据，系统自动使用默认值；
3. 数据库模式和内存模式结构一致。

### 8.3 前端壳测试

扩展 `test_frontend_shell.py`，验证：

- `/admin` 页面新增文档元数据显示区；
- `admin.js` 包含对应渲染字段。

### 8.4 回归

继续运行：

```bash
pytest tests/test_frontend_shell.py -v
pytest tests/test_document_api.py -v
pytest tests/test_database_mode_api.py -v
pytest -q
```

## 9. 验收标准

完成后应满足：

1. 文档模型新增 5 个元数据字段；
2. 上传接口支持写入元数据；
3. 未传元数据时有默认值；
4. 文档详情接口返回元数据字段；
5. `/admin` 文档详情区展示这些字段；
6. 测试补齐；
7. API 文档和技术映射文档同步更新；
8. 完整回归通过。

---

## 10. 建议结论

本阶段是“权限演进的地基”，不是“完整权限过滤实现”。

因此建议严格控制范围：

- 先把字段落库；
- 先把元数据带到上传、详情、后台；
- 不要在这一阶段把用户知识视图规则和检索过滤一次性做重。

这样可以保证后续每一步都清晰可控，也符合当前项目“一步一步来”的节奏。
