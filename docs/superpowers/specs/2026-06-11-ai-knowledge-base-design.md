# AI 知识库设计规格

- 版本：V1.0.0
- 目标上线时间：2026 年 8 月
- 最后更新：2026-06-11

## 1. 产品概述

### 1.1 定位

AI 知识库是一个面向公司内部人员及未来客户的 AI 原生知识库系统。基于公司内部所有产品相关文档，构建知识库并进行文档管理，提供即时、可溯源、权限安全的自然语言问答。后续计划推向化工、安保等垂直行业客户，实现知识库产品商业化。

### 1.2 核心价值

- 对内部使用人员：快速获取所需知识，提升工作效率
- 对产品部人员：降低产品/方案/技术类咨询量，释放人力
- 对公司发展：系统化知识沉淀与传承，自动发现知识缺口，提升新人培养效率

## 2. 总体架构

### 2.1 分层架构

```
用户层       Web 浏览器 (Vue 3 + Element Plus)       企微插件 (H5 嵌入)
              │                                         │
应用层 ┌──────┴─────────────────────────────────────────┴──────┐
       │  问答对话框   │  知识库浏览   │  管理后台（统计/配置）  │
       └──────┬─────────────────────────────────────────┬──────┘
              │                                         │
AI 层   ┌─────┴─────────────────────────────────────────┴─────┐
       │   LLM (Qwen/GPT-4/Claude)   │   Embedding (BGE/M3E)  │
       │   Reranker                  │   路由/融合层           │
       └─────┬───────────────────────────────────────────────┘
              │
检索层  ┌─────┴───────────────────────────────────────────────┐
       │  向量检索 (Milvus) — 一期上线                       │
       │  图检索 GraphRAG (Neo4j) — 二期叠加                  │
       │  全文关键词搜索                                     │
       └─────┬───────────────────────────────────────────────┘
              │
知识层  ┌─────┴───────────────────────────────────────────────┐
       │  文档存储 (原始文件)                                 │
       │  文本分块 (Chunks + Vectors)                        │
       │  知识图谱 (实体/关系，二期开始构建)                    │
       └─────┬───────────────────────────────────────────────┘
              │
接入层  ┌─────┴───────────────────────────────────────────────┐
       │  文档解析器 (Unstructured.io/LlamaParse)             │
       │  多模态抽取器 (LLM 驱动，渐进增强)                    │
       │  数据清洗   │   本体管理                              │
       └─────────────────────────────────────────────────────┘
```

### 2.2 架构原则

1. **渐进增强**：一期向量检索快速上线 MVP，二期叠加知识图谱增强，三期完善多模态和复杂场景
2. **混合模式**：知识图谱与向量检索并存，由路由层决定走哪个管道或合并结果
3. **向上兼容**：检索层接口抽象设计，KG 的加入不破坏上层逻辑
4. **知识驱动**：结构化知识（KG）+ 非结构化知识（向量）互为补充

## 4. 知识图谱设计

### 4.1 设计原则

- **一期只定义实体类型骨架**：不追求完备属性，只覆盖核心实体和关系
- **二期加属性和规则**：随使用数据完善本体
- **三期加动态约束**：冲突检测规则、时效性规则

### 4.2 核心本体骨架（一期定义，二期正式部署）

实体类型：

| 实体类型 | 说明 | 示例 |
|---------|------|------|
| `Product` | 产品/终端硬件设备 | P368 对讲机、调度台 |
| `Feature` | 功能特性 | SOS 报警、电子围栏、定位 |
| `Solution` | 解决方案 | 安防行业方案、工地调度方案 |
| `Document` | 文档/资料 | P368 用户手册 V2.3、产品白皮书 |
| `Parameter` | 技术参数 | 并发 1000、IP54 防护、群组 500 人 |
| `Scenario` | 应用场景 | 工地调度、酒店安保、化工巡检 |

关系类型：

| 关系类型 | 起点 | 终点 | 说明 |
|---------|------|------|------|
| `has_feature` | Product | Feature | 产品具有某个功能 |
| `has_parameter` | Product | Parameter | 产品具有某技术参数 |
| `belongs_to` | Product | Solution | 产品归属于某解决方案 |
| `applies_to` | Solution | Scenario | 解决方案适用于某场景 |
| `describes` | Document | Product | 文档描述某个产品 |
| `references` | Document | Solution | 文档引用某解决方案 |
| `depends_on` | Feature | Feature | 功能依赖另一功能 |
| `has_version` | Document | — | 文档版本属性（标注在节点上） |

### 4.3 本体管理方式

- 本体 Schema 在 Neo4j 中以约束 + 索引定义
- 管理员通过管理后台 UI 进行实体类型的增删改
- LLM 辅助抽取时回归到遵循当前 Schema

### 4.4 多模态抽取与实体关联

| 模态 | 一期（P0） | 二期（P1） | 三期（P2） |
|------|-----------|-----------|-----------|
| 文本 | 标题/段落解析，简单文本三元组抽取 | 多格式文档深度抽取 | 在线文档实时更新抽取 |
| 表格 | 简单表格转 Markdown 向量化 | 复杂表格结构还原 → 批量三元组 | — |
| 图片 | 仅作为附件保留 | OCR 文字识别 → 图片-实体关联 | 产品外观图 → 型号关联 |
| 流程图 | — | — | 图拓扑解析 → 流程三元组 |
| 语音 | — | — | 语音输入转文本查询 |

### 4.5 表格解析策略（解决 Dify PoC 的复杂表格痛点）

- **一期（简单表格）**：Excel/Word 中纯文本表格 → 转为 Markdown 表格格式 → 向量化
- **二期（复杂表格）**：使用多模态 LLM 识别表格结构 → 还原行列映射 → 批量生成三元组
  - 合并单元格识别
  - 表头-数据列映射
  - 每行转为 `(产品型号, 参数名, 参数值)` 结构

### 4.6 数据清洗

- 排版混乱文档上传时 → LLM 重新排版 → 结构化输出
- 清洗后规范文本展示给管理员确认 → 自动生成 QA 知识对 → 一键同步到知识库
- 重复文档检测：文档 hash + 语义相似度（余弦相似度 > 0.95 判定为重复）

## 5. 知识图谱维护机制

### 5.1 外部增量更新流程

```
新文档上传 → 文档解析/清洗 → 实体关系抽取（LLM）
    ↓
冲突检测（与已有 KG 比对）
    ├─ 无冲突 → 直接写入 KG
    ├─ 冗余（已有相同三元组）→ 跳过，记录来源
    ├─ 冲突（同一实体/关系/属性不同值）→ 进入人工审核队列
    └─ 过时（时间戳更早的同实体信息）→ 标记旧数据，写入新数据
    ↓
向量增量更新（仅重编变更文档分块，<5 分钟）
```

**冲突自动解决规则：**

1. **时间戳优先**：带明确时间的信息，新者胜
2. **来源优先级**：官方文档 > FAQ > 二手整理 > 抽取推测
3. **不确定性标记**：置信度低于阈值的三元组带 `uncertain` 标签

### 5.2 内部质量自维护（定时任务，二期+）

```
定时触发（如每周一次）
    ↓
扫描全量 KG
    ├─ 冗余检测：同实体同关系多个三元组 → 合并/去重
    ├─ 冲突检测：属性值不一致 → 按优先级规则解决 / 标记待审
    ├─ 过时检测：超过 TTL 的三元组 → 标记过期，排除出检索
    └─ 孤立实体：无关系的节点 → 标注待人工确认后归档
    ↓
维护日志写入，供管理员查看
```

- **原则**：默认只标记和预警，不自动删除，所有变更需管理员确认
- 过时判断基于文档最后更新时间 + 三元组置信度衰减 + 版本号

## 6. GraphRAG 图检索问答

### 6.1 混合检索架构

```
用户问题
    ↓
问题分析（LLM）
    ├─ 实体识别：提取产品名、功能名、参数名
    ├─ 意图分类：查参数？查功能？查方案？查文档？
    └─ 路由判断 → 走向量通道 / 走图通道 / 融合通道
                          │                  │
                   向量检索通道          图检索通道
                   1. 问题向量化          1. KG 中定位实体节点
                   2. 语义 Top-K          2. 1-3 跳邻居展开
                   3. Reranker 精排       3. 提取子图+路径
                          │                  │
                          └──────┬───────────┘
                                 ↓
                          LLM 生成答案 + 溯源
```

### 6.2 检索策略三阶段

| 阶段 | 检索方式 | 覆盖问题类型 | 对应产品期 |
|------|---------|-------------|----------|
| Phase 1 | 纯向量 RAG | 简单查询：找文档、查参数、FAQ | P0 |
| Phase 2 | 向量 + GraphRAG 并行合并 | 需整合推理：终端推荐、方案匹配 | P1 |
| Phase 3 | 智能路由 + 多 hop 图推理 | 复杂约束：多条件终端推荐、流程推理 | P2 |

### 6.3 GraphRAG 示例流程

```
问题："客户预算 5000 元，需要带摄像头和定位功能的终端"

KG 检索步骤：
  1. 找具有 Feature="摄像头" 的 Product → [A, B, C]
  2. 找具有 Feature="定位" 的 Product → [B, C, D]
  3. 求交集 → [B, C]
  4. 提取 B, C 的 has_parameter → 获取价格参数
  5. 过滤 Parameter ≤ 5000 → [B]
  6. 输出 B + 关联功能详情 + 来源文档
```

### 6.4 回答约束（幻觉控制）

- **无依据拒答**：检索相关度低于阈值（余弦相似度 < 0.6）→ 预设话术
  > "抱歉，在现有知识库中未找到相关依据，已通知管理员补充。"
- **推送通知**：拒答同时推送消息给管理员
- **低分答案处理**：用户反馈"没帮助" → 记录 → 管理员审阅 → 调优

```

    ↓ 点"没帮助" + 输入原因
    ↓

    ↓（管理员每周审阅）

    ├─ 文档有误 → 更新原文档 → 重新同步向量库

    ↓
知识库更新 → 下次相同问题自动改善
```

## 11. 非功能性需求

### 11.1 性能指标

| 指标 | 目标 | 实现方式 |
|------|------|---------|
| 检索响应 | <500ms | Milvus 索引优化 + Neo4j 索引 + Redis 缓存 |
| 首 token 延迟 | <1.5s | 流式输出 + LLM 预热 + 推理缓存 |
| 向量更新延迟 | <5min | Celery 异步任务批量处理 |
| 文档容量 | 50 万+ 片段 | Milvus 水平分片 + Neo4j 集群 |

### 11.2 知识新鲜度

- 文档 `last_updated` 时间戳跟随存储
- 检索时时间加权排序：越新权重越高
- 向量增量更新仅处理变更文档分块

### 11.3 安全性

- 知识库级别读写权限（LDAP/OAuth 集成可后续补充）
- 文档级别访问控制（二期细化）
- 知识库可见范围：部门/项目组/个人
